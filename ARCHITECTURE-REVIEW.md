# 架构评审（Architecture Review）

> 日期：2026-07-09 ｜ 评审对象：3d-game-kit 全仓库 ｜ 方法：对照三个基准
> ① 业主核心诉求：模块化、分层清晰、数据↔逻辑解耦、业务逻辑↔引擎解耦、明确"代码 vs 数据"边界
> ② 行业标准模式：Unity / Unreal / Godot / Bevy 中的等价物
> ③ 代码事实：以仓库当前实现为准，不以愿望为准

## 2. 七个需要诚实指出的架构真相

### 2.1 我们的"ECS"其实不是 ECS
现在的模型是 **Unity 经典 GameObject/MonoBehaviour 式的实体-组件（EC）**：组件持有逻辑+状态、每实体每帧 Update。真正的数据导向 ECS（Bevy/Unity DOTS）是组件=纯数据、系统=对组件查询做批处理。
**判定：当前选择是对的**（教学心智负担低、AI 语料多、规模用不到 DOTS 的性能），但两点要落实：① 文档改口叫"实体-组件（EC）架构"避免误导；② 知道天花板在哪——上万实体、并行模拟时才需要真 ECS，届时是重写不是演进。

### 2.2 "数据驱动"只覆盖了一半：实体的"组成"仍是代码
关卡摆放、任务、部分规则已是数据；但**"玩家=哪些组件+什么参数""警察=什么模型+什么速度"仍写死在 main.ts 的组装代码里**。行业里这一层叫 **Prefab / Archetype / Scene 文件**——是 Unity/Godot 最核心的资产形态，我们完全缺失。这是"游戏=引擎+数据"愿景里最大的一块空洞：换一个敌人类型、加一辆不同的车，现在都要改代码。

### 2.3 调参数值散落在代码常量里——正是业主点名的问题
`MAX_SPEED=17`、`PER_STAR=2`、`ESCAPE_DIST=45`、`CULL_DIST=65`、`fireRate=0.1`……全在组件文件顶部。它们的性质是**策划数值**（高频调整、不改行为结构），行业对应 ScriptableObject / DataTable。该进内容层的 `tuning` 数据，接受 L0 校验（范围检查）+ L1 模拟（数值合理性，如"车必须比警察快"——这条要是早就有，警察追车永不放弃的设计缺陷会在保存时被拦）。

### 2.4 事件机制有"两套半"，事件名有四处定义
- Entity.Broadcast（实体内部消息）
- demo 级 Bus（全局事件）
- window.__flight 旁路（黑匣子）
- rules.mjs 的 EVENTS 数组（第四处事件名清单）
行业做法：**引擎级类型化事件总线 + 单一事件注册表**。现状的风险：新事件名拼错无人拦、发了没人听静默失败、AI 改代码时不知道去哪找"合法事件列表"。

### 2.5 系统层直接摸 DOM 和渲染——阻断了"整游戏无头运行"（最重要的一条）
`WantedSystem.toast` 直接查 DOM；`PoliceNPC` 直接 `scene.add`；HUD 更新混在系统里。这就是**模拟/呈现未分离**（industry: simulation / presentation split）。后果具体而昂贵：
- L1 模拟器现在只能推演"规则抽象"，**不能跑真实系统**（WantedSystem 在 Node 里会因 DOM 崩掉）
- 通缉逃逸那 6 个 bug 的调试之所以要一轮轮开浏览器，正是因为系统没法无头跑
- 未来做联机（服务器权威模拟）、做录像回放、做 CI 全量逻辑测试，全部卡在这一条上
修复模式行业早有定论：**系统只改状态+发事件；View 层（HUD/toast/特效）订阅事件呈现**。

### 2.6 编辑三入口没有共享"变更实现"
CLI 直接改 JSON 文件、编辑器改内存再序列化、AI 走 CLI——**三个入口共享校验，却各写各的修改逻辑**。行业做法是**命令模式（Command Pattern）**：所有变更收敛为命令对象（`MoveMission{index,pos}`），一份实现、三个客户端，顺带免费获得 undo/redo（编辑器现在没有撤销，这在真实使用中很快会被骂）。

### 2.7 military-tps 是"框架成色"的照妖镜——目前照出的是私产
内容包、规则层、飞行记录仪、编辑器、L1 模拟……**全部长在 gta-sandbox 里**，military-tps 一项都没接入。检验"中间件/框架"是否成立的唯一标准是：**第二个游戏接入这些能力的成本**。现在答案是"要复制粘贴"，说明这些还是 gta 的功能，不是引擎的能力。

## 3. 目标分层（Target Architecture）

```
【第4层 内容 Content】   scene / missions / rules / tuning / prefabs   ← 全 JSON，全校验
【第3层 规则 Rules】     ECA 事件-条件-动作，封闭词汇表，可抽象模拟
【第2层 系统 Systems】   TS 组件：只改状态+发事件，无 DOM 无渲染依赖 ← 可无头
【第1.5层 呈现 View】    HUD / toast / 模型与特效挂接：订阅事件与状态
【第1层 引擎 Engine】    EC 内核 / 物理封装 / 输入 / 事件总线 / 时钟
【第0层 平台 Platform】  Three.js / Rapier / 浏览器 —— 上游维护，永不自研
─────────────────────────────────────────────────
横切能力（中间件包，所有游戏共享）：
  验证金字塔 L0/L1/L2 ｜ 飞行记录仪 ｜ 编辑命令层（CLI/编辑器/AI 三客户端）
```

**依赖铁律**：只准向下依赖；内容与规则层不含代码；系统层不摸 DOM/渲染；View 不含玩法决策；demo 之间零依赖，可共享物上提为包。

## 4. "代码 vs 数据"判定表（Rubric）

| 这东西的特征 | 归属 | 例子 |
|---|---|---|
| 连续实时行为、定义"手感" | **代码**（系统层） | 角色控制器、相机、车辆物理积分 |
| 高频改动、策划想调 | **数据**（tuning） | 车速、警力、射速、甩脱距离 |
| 空间坐标 | **数据**（命名点位） | 出生点、任务点、传送目标——**永不进代码**（已是铁律，血的教训） |
| 事件后果、触发式逻辑 | **数据**（规则层） | 被捕后干什么、几星时提示什么 |
| 实体的组成与配置 | **数据**（prefab，待建） | "警察=士兵模型+蓝色+追击组件(speed:4.4)" |
| 全新机制原型 | **先代码后固化** | 跑通后把可变部分抽成 tuning/规则动作，代码只留骨架 |

判定口诀：**"改它需要重新理解程序流程吗？"** 需要→代码；不需要（只是换数/换点/换后果）→数据。

## 6. 明确不做清单（架构纪律）

1. **不迁移数据导向 ECS**——教学场景收益为负，规模到达前不重写
2. **不自研渲染/物理/音频**——上游（Three.js/Rapier）维护，纪律不变
3. **规则层防图灵完备膨胀**——词汇表新增动作需评审"能否用现有动作组合"；条件表达式保持有限枚举，**不引入通用脚本表达式**；复杂到需要循环/变量的逻辑归还系统层。防止造出"一门更烂的编程语言"（Unreal Blueprint 面条图之鉴）
4. **JSON Schema 正式化暂缓**——手写校验器语义能力更强（空间检查），等编辑器需要字段级自动补全时再补 Schema
5. **联机暂不设计只留缝**——R1（事件化）+R5（确定性回放）天然是未来联机的地基，不提前建楼

## 7. 协作模式（对"我提一下你做一下"的回应）

本文件的 R1-R7 即 backlog。默认协作方式改为：**按序主动推进，每项按"实现→三级验证→commit→简报"交付，只在方向分叉点请业主拍板**。业主随时可重排优先级或叫停。
